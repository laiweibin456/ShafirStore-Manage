# 莎菲尔菓子门店管理系统项目开发框架

## 一、项目概述

### 1.1 项目目标

本项目旨在为莎菲尔菓子烘焙门店设计并实现一套轻量化的门店管理系统，采用 Spring Boot + Vue.js + MySQL 技术栈，实现商品管理、库存管理、销售管理、会员管理、员工权限管理及统计分析等核心功能，解决传统管理模式效率低下、数据分散、决策滞后等问题。

### 1.2 项目定位

- **用户群体**：中小型烘焙门店（莎菲尔菓子）
- **系统类型**：前后端分离的 B/S 架构管理系统
- **核心价值**：轻量化、高适配性、易操作

---

## 二、项目架构设计

### 2.1 整体架构

采用 **B/S 模式三层架构**，实现前端表现层、后端业务逻辑层、数据存储层的分离：

```
┌─────────────────────────────────────────────────────────────────┐
│                        客户端层（Client）                        │
│                    PC浏览器 / 平板设备                            │
├─────────────────────────────────────────────────────────────────┤
│                      前端表现层（View）                          │
│            Vue.js + HTML5 + CSS3 + JavaScript                  │
│      Element Plus + Vue Router + Pinia + Axios + ECharts       │
├─────────────────────────────────────────────────────────────────┤
│                      后端业务逻辑层（Service）                    │
│           Spring Boot + Spring Security + JWT                  │
│              MyBatis-Plus + Redis（可选缓存）                    │
├─────────────────────────────────────────────────────────────────┤
│                       数据存储层（Model）                          │
│                         MySQL 8.0                               │
│            商品表、库存表、订单表、会员表、员工表、角色表            │
└─────────────────────────────────────────────────────────────────┘
```

## 四、模块划分

### 4.1 后端模块划分

#### 4.1.1 认证模块（Auth Module）

**功能描述**：处理用户登录、登出、令牌刷新等认证相关操作

**核心接口**：

| 接口路径 | 请求方式 | 功能描述 |
|---------|---------|---------|
| `/api/auth/login` | POST | 用户登录 |
| `/api/auth/logout` | POST | 用户登出 |
| `/api/auth/refresh` | POST | 刷新令牌 |
| `/api/auth/current` | GET | 获取当前用户信息 |

**关键实现**：
- 使用 JWT 实现无状态认证
- 密码使用 BCrypt 加密存储
- 登录成功后返回 AccessToken 和 RefreshToken
- AccessToken 有效期 2 小时，RefreshToken 有效期 7 天

#### 4.1.2 用户权限模块（System Module）

**功能描述**：管理系统用户、角色、菜单和权限

**核心接口**：

| 接口路径 | 请求方式 | 功能描述 |
|---------|---------|---------|
| `/api/system/users` | GET | 用户列表查询 |
| `/api/system/users` | POST | 添加用户 |
| `/api/system/users/{id}` | PUT | 修改用户 |
| `/api/system/users/{id}` | DELETE | 删除用户 |
| `/api/system/roles` | GET | 角色列表查询 |
| `/api/system/roles` | POST | 添加角色 |
| `/api/system/roles/{id}` | PUT | 修改角色 |
| `/api/system/roles/{id}` | DELETE | 删除角色 |
| `/api/system/menus` | GET | 菜单列表查询 |
| `/api/system/logs` | GET | 操作日志查询 |

**角色划分**：

| 角色 | 权限范围 |
|------|---------|
| 店长 | 全功能操作，包含系统配置 |
| 店员 | 商品销售、库存查询、会员管理（无系统配置权限） |

#### 4.1.3 商品管理模块（Product Module）

**功能描述**：管理商品分类和商品信息

**核心接口**：

| 接口路径 | 请求方式 | 功能描述 |
|---------|---------|---------|
| `/api/products/categories` | GET | 分类列表查询 |
| `/api/products/categories` | POST | 添加分类 |
| `/api/products/categories/{id}` | PUT | 修改分类 |
| `/api/products/categories/{id}` | DELETE | 删除分类 |
| `/api/products` | GET | 商品列表查询 |
| `/api/products` | POST | 添加商品 |
| `/api/products/{id}` | GET | 商品详情查询 |
| `/api/products/{id}` | PUT | 修改商品信息 |
| `/api/products/{id}` | DELETE | 删除商品 |

**商品实体字段**：

| 字段 | 类型 | 说明 |

#### 4.1.4 库存管理模块（Inventory Module）

**功能描述**：管理库存入库、出库、预警等

**核心接口**：

| 接口路径 | 请求方式 | 功能描述 |
|---------|---------|---------|
| `/api/inventory` | GET | 库存列表查询 |
| `/api/inventory/{productId}` | GET | 单个商品库存查询 |
| `/api/inventory/in` | POST | 库存入库 |
| `/api/inventory/out` | POST | 库存出库 |
| `/api/inventory/records` | GET | 库存记录查询 |
| `/api/inventory/alerts` | GET | 低库存预警列表 |
| `/api/inventory/alerts/{id}` | PUT | 修改预警阈值 |

**库存联动逻辑**：
- 销售下单 → 自动扣减库存
- 库存不足 → 提示库存短缺
- 库存低于预警值 → 发送预警通知

#### 4.1.5 销售管理模块（Sale Module）

**功能描述**：处理收银下单、订单管理、销售统计

**核心接口**：

| 接口路径 | 请求方式 | 功能描述 |

| `/api/orders/{id}/refund` | PUT | 订单退款 |


- **后端权限控制**：基于 Spring Security + JWT 实现接口级别权限控制
- **前端权限控制**：基于用户角色动态渲染菜单和按钮
- **敏感操作日志**：记录所有关键操作的详细信息

### 6.2 库存销售联动模块

#### 6.2.1 销售下单流程

```
1. 收银员选择商品添加到购物车
         ↓
2. 系统校验库存是否充足
    ├── 库存充足 → 继续下一步
    └── 库存不足 → 提示缺货，阻止下单
         ↓
3. 会员识别（扫描会员码或输入手机号）
    ├── 会员识别成功 → 计算积分和优惠
    └── 非会员 → 按普通价格计算
         ↓
4. 确认订单信息，选择支付方式
         ↓
5. 支付成功后：
    ├── 库存扣减（异步或同步）
    ├── 积分增加（如果是会员）
    ├── 生成销售订单
    └── 打印小票（可选）
         ↓
6. 返回订单完成信息
```

#### 6.2.2 库存预警机制

- 库存低于设定阈值时，系统自动提醒
- 支持按商品分类设置不同的预警值
- 首页展示库存预警信息

### 6.3 会员积分模块

#### 6.3.1 积分计算规则

| 操作类型 | 积分规则 |
|---------|---------|
| 消费积分 | 每消费1元 = 1积分（可配置） |
| 积分抵扣 | 100积分 = 1元（可配置） |
| 生日双倍 | 会员生日当月消费双倍积分 |

#### 6.3.2 会员等级规则

| 等级 | 累计消费金额 | 享受折扣 |
|------|-------------|---------|
| 普通会员 | 0 - 500元 | 无 |
| 银卡会员 | 500 - 2000元 | 9.5折 |
| 金卡会员 | 2000 - 5000元 | 9折 |
| 钻石会员 | 5000元以上 | 8.5折 |

## 八、开发规范

### 8.1 编码规范

#### 8.1.1 后端编码规范

- **命名规范**：
  - 类名：UpperCamelCase（如 `UserController`）
  - 方法名：lowerCamelCase（如 `getUserById`）
  - 常量：UPPER_SNAKE_CASE（如 `MAX_RETRY_COUNT`）
  - 数据库表名/字段名：snake_case（如 `user_name`）

- **Controller层规范**：
  - 每个模块一个 Controller
  - 方法添加清晰的注释
  - 参数使用实体类封装
  - 返回统一的 Result 对象

- **Service层规范**：
  - 接口与实现分离
  - 业务逻辑事务控制
  - 异常统一处理

- **Mapper层规范**：
  - MyBatis-Plus 常用方法直接使用
  - 复杂SQL编写在XML文件中
  - SQL语句需要注释说明

#### 8.1.2 前端编码规范

- **目录规范**：
  - 按模块划分目录
  - 组件使用 PascalCase 命名
  - 工具函数使用 camelCase 命名

- **组件规范**：
  - 组件文件不超过500行
  - 样式使用 SCSS 预处理器
  - 状态管理按模块划分

- **API规范**：
  - 统一封装 Axios 请求
  - API 方法按模块划分文件
  - 请求参数和响应数据有 TypeScript 类型定义（可选）

### 10.3 部署步骤

#### 10.3.1 后端部署

```bash
# 1. 打包项目
mvn clean package -DskipTests

# 2. 上传JAR包到服务器
scp target/shafir-store.jar user@server:/opt/shafir-store/

# 3. 创建启动脚本
vim start.sh
#!/bin/bash
java -jar /opt/shafir-store/shafir-store.jar --spring.profiles.active=prod

# 4. 启动服务
chmod +x start.sh
./start.sh
```

#### 10.3.2 前端部署

```bash
# 1. 设置生产环境API地址
vim .env.production
VITE_API_BASE_URL=https://api.shafirstore.com

# 2. 打包项目
npm run build

# 3. 上传静态资源到Nginx
scp -r dist/* user@server:/usr/share/nginx/html/

# 4. 配置Nginx
vim /etc/nginx/conf.d/shafirstore.conf
```

#### 10.3.3 Nginx 配置示例

```nginx
server {
    listen 80;
    server_name shafirstore.com;

    # 前端静态资源
    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # 后端API代理
    location /api {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```
