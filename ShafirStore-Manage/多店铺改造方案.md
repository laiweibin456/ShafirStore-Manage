# 第一部分：需求分析

## 1.1 项目背景与目标

### 1.1.1 现状分析

ShafirStore（莎菲尔菓子门店管理系统）当前为**单店铺架构**，存在以下核心限制：

| 限制项 | 现状 | 影响 |
|--------|------|------|
| 数据模型无店铺维度 | 全部13张业务表均无 `store_id` 字段 | 无法区分数据归属店铺 |
| 权限体系单层 | 仅区分"已登录/未登录"，无角色级别控制 | 无法限制用户访问范围 |
| 会员体系单店铺 | 会员消费/积分/等级绑定在单一维度 | 多店铺无法共享会员 |
| 统计分析全局化 | 无店铺维度筛选 | 无法对比各店铺经营状况 |
| 小程序固定店铺 | API地址硬编码，无店铺选择 | 无法服务多店铺客户 |

### 1.1.2 改造目标

| 目标 | 描述 | 优先级 |
|------|------|--------|
| 多店铺数据隔离 | 各店铺商品、库存、订单、积分数据完全隔离 | P0 |
| 统一管理平台 | 超级管理员可管理所有店铺，查看跨店铺汇总数据 | P0 |
| 会员跨店铺共享 | 会员基本信息全局共享，消费/积分/等级按店铺独立 | P0 |
| 细粒度权限控制 | 三级角色体系，支持店铺级权限隔离 | P0 |
| 小程序多店铺 | 支持店铺选择、按店铺浏览商品和下单 | P1 |
| 跨店铺数据分析 | 各店铺销售对比、趋势分析、经营排行 | P1 |
| 现有业务不中断 | 改造过程中现有功能正常运行 | P0 |

## 1.3 非功能需求

### 1.3.1 性能需求

| 编号 | 需求 | 指标 | 验收标准 |
|------|------|------|---------|
| NFR-001 | 接口响应时间 | 商品列表查询 < 200ms | 1000条商品数据下P95 < 200ms |
| NFR-002 | 订单创建TPS | > 50 TPS | 10个店铺并发创建订单 |
| NFR-003 | 统计查询响应 | < 500ms | 10万订单数据量下 |
| NFR-004 | 店铺切换延迟 | < 300ms | 含Token重新生成 |
| NFR-005 | 租户拦截器开销 | SQL执行时间增加 < 5% | 对比无拦截器基准 |
| NFR-006 | 并发用户数 | 支持100并发用户 | 响应时间不超标的最大并发 |

### 1.3.2 安全需求

| 编号 | 需求 | 描述 | 验收标准 |
|------|------|------|---------|
| NFR-010 | 数据隔离 | 不同店铺数据严格隔离 | 店铺A用户无法访问店铺B数据 |
| NFR-011 | 密码安全 | 使用BCrypt加密存储 | 数据库中无明文密码 |
| NFR-012 | 接口鉴权 | 所有写操作需认证 | 未认证请求返回401 |
| NFR-013 | 权限最小化 | 用户仅能访问授权店铺数据 | 越权访问返回403 |
| NFR-014 | Token安全 | JWT Token包含店铺信息且不可篡改 | 篡改storeId后Token验证失败 |
| NFR-015 | 文件上传安全 | 上传接口需认证+文件类型校验 | 未认证上传返回401 |

### 1.3.3 可用性需求

| 编号 | 需求 | 描述 | 验收标准 |
|------|------|------|---------|
| NFR-020 | 向后兼容 | 旧版Token在过渡期可用 | 旧Token可正常访问默认店铺 |
| NFR-021 | 平滑迁移 | 迁移过程不停机超过30分钟 | 停机窗口 < 30分钟 |
| NFR-022 | 数据完整性 | 迁移后数据无丢失 | 各表记录数与迁移前一致 |

### 1.3.4 可扩展性需求

| 编号 | 需求 | 描述 | 验收标准 |
|------|------|------|---------|
| NFR-030 | 店铺数量 | 支持100+店铺 | 100店铺下性能不降级 |
| NFR-031 | 数据量 | 单店铺10万订单 | 查询性能不降级 |
| NFR-032 | 水平扩展 | 支持应用层水平扩展 | 无状态设计，可部署多实例 |

### 1.4.2 核心用户场景

**场景1：超级管理员创建新店铺**

```
前置条件：已登录超级管理员账号
1. 进入"店铺管理"页面
2. 点击"新增店铺"
3. 填写店铺信息（名称、编码、地址、电话、营业时间、Logo）
4. 点击"保存"
5. 系统创建店铺记录
6. 系统自动创建该店铺的默认会员等级（普通/银卡/金卡/钻石）
7. 提示"店铺创建成功"
后置条件：新店铺可用，可分配管理员
```

**场景2：店铺管理员切换店铺**

```
前置条件：已登录，关联了多个店铺
1. 点击顶部栏店铺选择器
2. 下拉列表显示有权限的店铺
3. 选择"莎菲尔菓子(春熙路店)"
4. 系统调用 POST /api/auth/switch-store
5. 后端验证用户有该店铺权限
6. 重新生成JWT Token（storeId=新店铺ID）
7. 前端更新Token和店铺上下文
8. 页面数据刷新为新店铺的数据
后置条件：所有操作在新店铺上下文中执行
```

**场景3：会员在多店铺消费**

```
前置条件：会员"李女士"已在A店注册
1. 李女士在小程序切换到B店
2. 浏览B店商品并下单
3. 系统自动创建 member_store_rel（B店关联）
4. B店独立计算消费金额和积分
5. 李女士在A店的积分/等级不受影响
6. 李女士可在"我的"页面查看各店铺积分
后置条件：会员在A店和B店有独立的积分和等级
```

**场景4：超级管理员查看跨店铺统计**

```
前置条件：已登录超级管理员账号
1. 进入"总控仪表盘"
2. 查看全店铺汇总数据（总销售额、总订单数、总会员数）
3. 查看各店铺销售对比柱状图
4. 查看各店铺销售趋势折线图
5. 点击"店铺经营排行"查看TOP店铺
6. 可选择时间范围筛选
后置条件：数据实时反映各店铺经营状况
```

**场景5：店员在收银台操作**

```
前置条件：已登录店员账号，当前店铺为A店
1. 进入收银台页面
2. 扫描/搜索商品（仅显示A店商品）
3. 添加商品到购物车
4. 选择会员（仅显示A店会员或在A店消费过的会员）
5. 选择支付方式
6. 确认收款
7. 系统创建A店订单，扣减A店库存
8. 会员在A店的积分/消费额更新
后置条件：订单、库存、积分均在A店维度正确更新
```

## 2.5 安全策略

### 2.5.1 认证体系

```
┌─────────────────────────────────────────────────────────────┐
│                     认证流程                                  │
│                                                              │
│  1. 用户登录                                                 │
│     POST /api/auth/login {username, password}                │
│     ├── BCrypt密码验证（替换明文比对）                         │
│     ├── 查询用户关联的店铺列表                                 │
│     ├── 默认选择第一个店铺                                     │
│     └── 生成JWT Token (含storeId/storeIds)                   │
│                                                              │
│  2. 请求认证                                                 │
│     Authorization: Bearer <token>                            │
│     ├── JwtAuthFilter解析Token                               │
│     ├── 提取userId/username/role/storeId/storeIds            │
│     ├── 构建SecurityUser对象                                  │
│     ├── 设置StoreContext(ThreadLocal)                         │
│     └── 放入SecurityContextHolder                            │
│                                                              │
│  3. 店铺切换                                                 │
│     POST /api/auth/switch-store {storeId}                    │
│     ├── 验证storeId在用户storeIds列表中                       │
│     ├── 重新生成Token (新storeId)                             │
│     └── 返回新Token                                          │
│                                                              │
│  4. 请求结束                                                 │
│     StoreContextFilter.clear()  ← 清除ThreadLocal            │
└─────────────────────────────────────────────────────────────┘
```

## 3.2 索引优化

### 3.2.1 现有索引调整

| 表 | 旧索引 | 新索引 | 原因 |
|----|--------|--------|------|
| inventory | `UNIQUE(product_id)` | `UNIQUE(store_id, product_id)` | 同一商品在不同店铺有独立库存 |
| product | `INDEX(category_id)` | `INDEX(store_id, category_id)` | 按店铺+分类查询是高频场景 |
| sale_order | `INDEX(create_time)` | `INDEX(store_id, create_time)` | 统计查询按店铺+时间范围 |

### 3.2.2 新增索引

| 表 | 新索引 | 用途 |
|----|--------|------|
| sale_order | `INDEX(store_id, status, create_time)` | 订单列表按店铺+状态+时间查询 |
| sale_order_item | `INDEX(store_id, product_id, create_time)` | 商品销售排行按店铺统计 |
| inventory_record | `INDEX(store_id, create_time)` | 库存记录按店铺+时间查询 |
| member_store_rel | `INDEX(store_id, level)` | 按店铺查询各等级会员数 |
| member_store_rel | `INDEX(store_id, total_consume)` | 按店铺查询消费排行 |

### 3.2.3 索引设计原则

1. **最左前缀原则**：`store_id` 作为联合索引的第一列，确保租户隔离查询命中索引
2. **覆盖索引**：统计查询尽量使用覆盖索引，避免回表
3. **避免冗余**：`INDEX(store_id)` 与 `INDEX(store_id, create_time)` 共存时，前者可删除
4. **索引数量控制**：单表索引不超过6个
